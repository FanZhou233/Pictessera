Pictessera Photos 图片管理系统
现有功能、技术框架与自动分类需求说明
版本：1.0
日期：2026-07-09

======================================================================
一、项目定位
======================================================================

本系统是面向 Windows 的本地 iCloud 照片与实况照片管理工具。

核心目标：
1. 扫描本地导出的 iCloud 照片目录。
2. 将 HEIC/JPEG/PNG 静态图片与对应 MOV 组合为 Live Photo。
3. 提供高性能照片墙、表格、搜索、预览、筛选和批量文件操作。
4. 所有照片和元数据默认仅在本机处理，不上传到云端。
5. 在保留完整功能的前提下，提供轻量版与 Plus 完整播放版。


======================================================================
二、现有功能
======================================================================

2.1 文件扫描

- 支持选择本地源文件夹。
- 默认递归扫描子文件夹。
- 支持的图片格式：
  HEIC、HEIF、JPG、JPEG、PNG。
- 支持的动态组件格式：
  MOV。
- 扫描任务在后台线程执行，不阻塞主界面。
- 支持停止扫描和重新扫描。
- 显示扫描、缩略图和文件操作进度。


2.2 照片与 Live Photo 识别

- 根据文件名和文件关系组合静态图片与 MOV。
- 将组合后的图片作为一个 Live Photo 项目展示。
- 识别未找到对应静态图片的独立 MOV。
- 未绑定 MOV 可以由用户手动绑定到静态图片。
- 保存人工绑定关系，下次扫描可恢复。
- 支持检查原始文件是否仍然存在。


2.3 时间与元数据

- 读取 EXIF/HEIF 拍摄时间。
- 兼容常见时区、ISO 时间和 EXIF 时间格式。
- 元数据不可用时回退到文件时间。
- 支持按拍摄时间排序。
- 显示文件名、类型、拍摄时间、时间来源、路径、文件数量和大小。
- 详情窗口可显示格式化后的 EXIF、GPS 和文件信息。


2.4 缩略图与照片墙

- 使用 QListView + QAbstractListModel 构建虚拟化照片墙。
- 只为可见项目请求数据，适合浏览大量照片。
- 支持后台生成缩略图。
- 支持本地缩略图缓存。
- Live Photo 项目带有状态标识。
- 支持鼠标悬停预览 Live Photo。


2.5 表格视图

- 使用 QTableView + QAbstractTableModel 构建虚拟化表格。
- 支持照片墙与表格之间切换。
- 切换视图不会重新创建成千上万个 Qt 控件。
- 两种视图共享选择状态和当前筛选结果。


2.6 详情预览

- 双击照片打开非模态详情窗口。
- 支持同时打开多个详情窗口。
- 支持上一张、下一张导航。
- 支持图片缩放和拖拽查看。
- 支持 Live Photo 快速预览和高清预览。
- 支持在详情窗口中执行删除等操作。


2.7 搜索

- 支持在当前筛选结果中搜索。
- 支持普通包含搜索。
- 支持 Windows Explorer 风格通配符：
  *、?、[]。
- 支持使用分号或竖线分隔多个条件。
- 示例：
  *.HEIC
  IMG_12??
  *.MOV;*.HEIC
- 搜索字段包括文件名、路径、扩展名、媒体类型、时间来源和拍摄时间。
- 点击搜索结果可以定位并选中照片。


2.8 当前筛选分类

以下分类已经实现并可使用：

- 所有照片。
- 实况照片。
- 静态照片。
- 待绑定视频。
- 最近删除。

注意：
当前界面中的“智能分类”仅为说明区域，尚未实现自动分类逻辑。


2.9 选择与批量操作

- 支持单选、多选、全选和取消选择。
- 支持拖动范围选择。
- 支持右键菜单。
- 显示已选项目数量、文件数量、空间占用和选择比例。
- 支持批量移动。
- 支持按重新排列后的顺序导出。
- 支持按照 iOS DCF 目录和 IMG 编号规则生成导出结构。


2.10 删除、恢复和历史

- 删除项目时先进入应用内“垃圾箱”状态。
- 支持从垃圾箱恢复。
- 支持将垃圾箱项目移动到程序目录下的“已删除”文件夹。
- 支持清空垃圾箱。
- 支持撤销与重做。
- 使用日志和状态文件保存操作信息。


2.11 状态与缓存

- 程序状态保存在程序目录下的 Pictessera_Data。
- 主要状态包括：
  垃圾箱状态、垃圾箱日志、项目元数据缓存、MOV 绑定关系、
  撤销/重做会话和缩略图缓存。
- JSON 状态采用临时文件替换方式原子写入。
- 支持备份和损坏文件隔离。
- 使用稳定路径键和文件签名判断缓存是否仍然有效。


2.12 界面

- 使用 PySide6 构建 Windows 桌面界面。
- 当前主界面采用 iTunes 风格布局：
  顶部标题栏、资料库导航、左侧分类栏、右侧照片内容区。
- 使用自绘无边框窗口。
- 支持最小化、最大化、关闭和边缘缩放。
- 已接管 Windows 非客户区计算，最大化时不保留黑色边缘缝隙。
- 普通状态使用轻微圆角，最大化时取消圆角和外边距。


2.13 打包版本

轻量版：
- 配置文件：Pictessera.spec。
- 输出文件：dist/Pictessera.exe。
- 排除 OpenCV、NumPy、ImageIO 等大型可选依赖。
- 保留照片管理核心功能。
- 目标电脑没有 ffmpeg 时不保证 Live Photo 动态播放。

Plus 版：
- 配置文件：PictesseraPlus.spec。
- 输出文件：dist/Pictessera-Plus.exe。
- 包含 OpenCV、NumPy、ImageIO 和 imageio-ffmpeg。
- 内置 ffmpeg，支持 iPhone HEVC MOV 解码。
- 体积大于轻量版，但提供完整 Live Photo 播放回退链。


======================================================================
三、现有技术框架
======================================================================

3.1 运行环境

- Python 3.9。
- Windows 10/11。
- Conda base 为当前开发环境。


3.2 GUI

- PySide6 / Qt 6。
- QListView + QAbstractListModel。
- QTableView + QAbstractTableModel。
- 自定义 Delegate、滚动条、组合框、按钮、标题栏和窗口边框。
- Qt Signal/Slot 负责界面和后台任务通信。


3.3 图片与媒体

- Pillow：
  图片读取、裁剪、缩放、PNG 编码和 EXIF 处理。
- pillow-heif：
  HEIC/HEIF 解码。
- imageio-ffmpeg：
  Plus 版首选的 HEVC MOV 解码后端。
- OpenCV + NumPy：
  Plus 版 Live Photo 回退解码。
- ImageIO：
  Plus 版最后一级视频解码回退。


3.4 并发模型

- Python threading。
- 自定义 AppThreadPoolExecutor。
- 缩略图、元数据、Live Photo、详情图和文件操作使用独立线程池。
- 工作线程为 daemon，关闭程序时不会被慢任务长期阻塞。
- 使用 stop_event 和 generation 防止旧任务结果污染新扫描。


3.5 数据模型

核心领域对象：PhotoItem。

主要字段：
- item_id。
- display_name。
- files。
- size_bytes。
- representative_image。
- is_live。
- item_type。
- shot_time。
- time_source。
- source_folder。
- stable_key。
- file_signature。
- item_kind。
- bound_image_paths。
- needs_binding。


3.6 当前代码结构

main.py
- 兼容入口。
- 仍包含大部分 UI、媒体、状态和文件操作代码。

photo_manager/config.py
- 配置、主题、并发数量和媒体参数。

photo_manager/bootstrap.py
- Qt/Conda 启动兼容。
- PyInstaller 资源路径解析。

photo_manager/domain/
- 与 Qt 无关的领域模型。

photo_manager/services/
- 可复用业务服务。
- 当前已提取搜索服务。

photo_manager/infrastructure/
- 线程池等基础设施。

tests/
- 与 GUI 解耦的快速单元测试。

依赖方向：

main.py / UI
    -> services
    -> domain
    -> config

infrastructure
    -> domain / config

四、新需求：自动分类
======================================================================

4.1 需求目标

系统在扫描照片后，根据文件、EXIF 和媒体信息自动生成虚拟分类，
用户无需手工建立文件夹即可快速查找照片。

自动分类默认不移动、不复制、不重命名原始文件。
分类只保存照片与分类之间的关联关系。
一个照片可以同时属于多个分类。


4.2 第一阶段范围（MVP）

必须实现以下自动分类：

1. 按时间分类
   - 年份。
   - 年月。
   - 最近 7 天。
   - 最近 30 天。
   - 时间未知。

2. 按媒体类型分类
   - Live Photo。
   - 静态照片。
   - HEIC/HEIF。
   - JPEG。
   - PNG。
   - 独立或未绑定 MOV。

3. 按来源分类
   - 按源文件夹。
   - 按一级子目录。

4. 按拍摄设备分类
   - 相机品牌 Make。
   - 相机型号 Model。
   - 设备信息未知。

5. 按地点分类
   - 有 GPS 信息。
   - 无 GPS 信息。
   - 第一阶段只保存经纬度分组，不强制接入在线地图或地名 API。

6. 按文件特征分类
   - 小图片。
   - 大图片。
   - 大文件。
   - 元数据缺失。


4.3 可选第二阶段范围

- 按国家、省、市分类。
- 同一天或相近时间形成“事件”。
- 连拍照片识别。
- 屏幕截图识别。
- 相似照片识别。
- 重复照片识别。
- 模糊照片识别。
- 人脸聚类。
- 人物、宠物、食物、风景、文档等内容识别。
- 用户自定义规则。
- 收藏、评分和人工标签。

第二阶段涉及图像特征或 AI 模型，应作为可选 Plus/AI 模块，
不能强制增加轻量版体积。


4.4 分类原则

- 分类是虚拟视图，不直接修改原文件。
- 一个项目允许拥有多个分类标签。
- Live Photo 的图片和 MOV 必须作为同一个 PhotoItem 分类。
- 最近删除中的项目不进入普通智能分类统计。
- 恢复项目后自动恢复其分类。
- 未绑定 MOV 只进入视频相关分类。
- 拍摄时间优先使用已验证 EXIF 时间。
- 文件时间只能作为回退，并标记时间来源。
- 所有分类结果必须可重复计算。

左侧“智能分类”区域改为可用功能区。

建议层级：

  媒体类型
    实况照片
    静态照片
    HEIC
    JPEG
    PNG
  设备
    Apple iPhone ...
  地点
    有位置信息
    未知地点
  文件状态
    大文件
    元数据缺失

交互要求：
- 分类项显示照片数量。
- 点击分类后更新照片墙和表格。
- 当前分类有明确选中状态。
- 分类可以展开和折叠。
- 搜索只搜索当前分类结果。
- 所有照片、垃圾箱等现有分类继续保留。
- 扫描过程中允许逐步更新分类数量。
- 自动分类失败不能阻塞基础照片浏览。


4.6 数据结构建议

新增分类定义：

AutoCategory
- category_id：稳定唯一 ID。
- category_type：time、media、device、location、file 等。
- name：显示名称。
- parent_id：父分类 ID。
- sort_key：排序键。
- rule_version：规则版本。
- item_count：当前项目数量。

新增项目分类关系：

ItemCategoryRelation
- stable_key：PhotoItem 稳定键。
- category_id：分类 ID。
- classified_at：分类时间。
- source_signature：分类时的文件签名。

分类缓存建议文件：

Pictessera_Data/auto_categories.json
Pictessera_Data/item_category_relations.json

如果照片数量较大，后续可改用 SQLite：

Pictessera_Data/photo_manager.db


4.7 模块设计要求

自动分类代码不得继续直接写入 main.py。

建议新增：

photo_manager/domain/categories.py
- AutoCategory。
- ItemCategoryRelation。
- 分类枚举。

photo_manager/services/classification.py
- ClassificationService。
- 规则注册。
- 单项目分类。
- 批量分类。
- 增量分类。

photo_manager/services/classification_rules/
- time_rule.py。
- media_rule.py。
- device_rule.py。
- location_rule.py。
- file_rule.py。

photo_manager/infrastructure/category_repository.py
- 分类缓存读取。
- 原子保存。
- 缓存版本迁移。

UI 只负责：
- 显示分类树。
- 发送用户选择。
- 接收分类结果。
- 更新照片模型。


4.8 性能要求

- 扫描 10,000 个 PhotoItem 时，分类必须在后台执行。
- 分类过程中主界面保持可操作。
- 已缓存且文件签名未变化的项目不得重复读取完整 EXIF。
- 单个项目变化时只重新分类该项目。
- 分类结果分批提交给 UI，避免高频重建模型。
- UI 分类切换继续使用 visible_ids，不创建独立照片控件。
- 第一阶段规则分类不能依赖 GPU。
- 关闭程序时分类任务必须能够立即取消。


4.9 缓存失效规则

满足任一条件时重新分类：

- 文件路径变化。
- 文件大小变化。
- 修改时间变化。
- 文件签名变化。
- EXIF 元数据更新。
- 人工 MOV 绑定关系变化。
- 分类规则版本升级。

源文件被删除时：
- 删除对应分类关系。
- 更新分类数量。
- 不保留无效 stable_key。


4.10 错误处理

- 单张照片分类失败不得中断整个分类任务。
- 记录失败项目和失败原因。
- 无 EXIF、损坏图片和不支持格式进入相应异常分类。
- 分类缓存损坏时自动隔离并重新生成。
- 禁止因分类失败修改或删除原文件。


4.11 验收标准

1. 扫描完成后自动生成时间、媒体、设备、地点和文件状态分类。
2. 点击任一分类，照片墙和表格显示相同项目集合。
3. 同一 Live Photo 不得被拆成图片和 MOV 两项。
4. 同一照片可以同时出现在时间、设备和媒体分类中。
5. 重新启动程序后可以从缓存恢复分类。
6. 未变化照片不会重复执行昂贵的元数据读取。
7. 新增照片后只对新增或变化项目执行增量分类。
8. 删除、恢复、绑定 MOV 后分类数量立即正确更新。
9. 分类过程不阻塞滚动、选择、搜索和详情预览。
10. 10,000 项数据下分类界面仍可正常展开、切换和搜索。
11. 自动分类不移动、不复制、不重命名任何原始文件。
12. 轻量版可运行第一阶段规则分类，不增加大型 AI 依赖。

======================================================================
六、非目标
======================================================================

第一阶段明确不做：

- 不自动移动原始照片。
- 不自动删除重复照片。
- 不调用在线人脸或图像识别服务。
- 不强制上传照片、EXIF 或 GPS 信息。
- 不因自动分类改变现有垃圾箱和撤销/重做语义。
- 不将大型 AI 模型加入轻量版。
